OPENCLAW + N8N PROJECT CONTEXT
AI SHORT-FORM CONTENT FACTORY
MASTER PLAN (REVISED FOR N8N)

--------------------------------------------------
1. PROJECT GOAL
--------------------------------------------------

Build an automated AI short-form content system that is reliable enough to run daily with minimal manual work.

This system should automatically:

1) Collect or receive topic ideas
2) Generate viral topics
3) Generate hooks
4) Generate scripts
5) Generate voice (TTS)
6) Generate subtitles
7) Render short videos
8) Queue uploads
9) Publish videos to

- TikTok
- YouTube Shorts
- Instagram Reels


--------------------------------------------------
2. CORE OPERATING PRINCIPLE
--------------------------------------------------

This project is no longer OpenClaw-only.

The correct operating model is:

OpenClaw = builder / experiment lab / dev helper
n8n = runner / production pipeline / control tower
MLX or local Qwen = cheap local text engine for routine generation
GPT / Codex = premium reasoning, script improvement, coding, debugging
ffmpeg = deterministic render engine
ElevenLabs = production TTS

Important principle:

Deterministic workflow first.
Agentic autonomy second.

In other words:

Do not use OpenClaw as the main always-on production runner.
Use n8n as the main scheduler, router, queue manager, and execution controller.

--------------------------------------------------
13. N8N RESPONSIBILITIES
--------------------------------------------------

n8n is the production control tower.

n8n should handle:

1) Triggering workflows by schedule
2) Receiving manual commands from Telegram or webhook
3) Calling MLX or GPT endpoints for text generation
4) Calling shell scripts for TTS, subtitles, rendering, and publishing
5) Moving files between queue folders
6) Marking success / failure states
7) Sending daily reports
8) Sending failure alerts
9) Retrying failed jobs safely

n8n should NOT become the place where complex creative strategy is invented from scratch every run.
That belongs to prompts, templates, and builder workflows.


--------------------------------------------------
14. MLX RESPONSIBILITIES
--------------------------------------------------

MLX local server should be used for:

- topic expansion
- hook draft generation
- short script draft generation
- classification
- formatting
- rewriting into structured JSON

MLX should NOT be treated as a perfect production-grade public API.

Use MLX as:

- local internal engine
- low-cost generator
- speed layer for routine tasks

Use GPT / Codex when:

- script quality is weak
- prompt structure changes
- integration code is needed
- debugging is complex
- publishing logic breaks
- higher quality copy is required


--------------------------------------------------
15. OPENCLAW RESPONSIBILITIES AFTER N8N MIGRATION
--------------------------------------------------

OpenClaw remains important, but its job changes.

OpenClaw should now be used for:

- designing prompts
- generating or improving templates
- code writing
- workflow planning
- ops troubleshooting
- ad hoc batch content generation
- experimentation with new content formats

OpenClaw should not be the main repeating scheduler for the daily production line.

Simple rule:

OpenClaw builds the machine.
n8n runs the machine.


--------------------------------------------------
16. SECURITY POLICY
--------------------------------------------------

Security rules for this project:

1) Start with n8n accessible only inside local network or Tailscale
2) Do not expose n8n publicly until authentication, HTTPS, secret handling, and routing are clean
3) Keep MLX server local-only whenever possible
4) Store API keys outside prompts and outside versioned workflow exports
5) Use file permissions for secret files
6) Separate credentials from content data
7) Treat uploader credentials as production secrets

Recommended secret locations:

- ~/.openclaw/credentials
- n8n credentials store
- environment files
- mounted secret files

Never hardcode production keys inside prompts or shell scripts.

18. REQUIRED N8N WORKFLOWS
--------------------------------------------------

Minimum workflow set:

WF01_topic_intake
- Trigger: schedule or Telegram command
- Result: daily topic list JSON

WF02_hook_and_script_generation
- Input: topic JSON
- Engine: MLX first, GPT fallback when needed
- Result: hook JSON + script JSON

WF03_audio_generation
- Input: approved scripts
- Engine: ElevenLabs
- Result: mp3 files

WF04_subtitle_generation
- Input: scripts or TTS transcript
- Result: srt files

WF05_video_render
- Input: script + mp3 + srt + character image
- Engine: ffmpeg via shell script
- Result: mp4 files

WF06_publish_queue_builder
- Input: rendered mp4
- Result: queue entries for platforms

WF07_platform_uploader
- Input: queue items
- Result: upload attempt + success/fail log

WF08_daily_report
- Output: counts, failures, uploaded videos, pending items

WF09_failure_watch_and_retry
- Output: retries, alerts, failed item isolation

--------------------------------------------------
21. IMPLEMENTATION ROADMAP
--------------------------------------------------

STEP 0
FREEZE CURRENT WORKING ASSETS

Do not delete current outputs.
Back up:

- topics
- hooks
- scripts
- pipeline prompt files
- existing shell scripts


STEP 1
INSTALL N8N AS THE RUNNER

Target:
- self-hosted n8n via Docker
- internal-only access at first
- reachable through local machine or Tailscale

Goal:
- one stable workflow runner
- one place for schedule, retry, reporting, and queue logic


STEP 2
RUN MLX AS LOCAL TEXT SERVER

Target:
- local-only endpoint
- used by n8n HTTP Request nodes
- used for cheap routine generation

Goal:
- reduce paid API dependence for repetitive tasks


STEP 3
CONNECT N8N TO EXISTING CONTENT FACTORY

n8n should call existing local scripts and write into:

- topics
- hooks
- scripts
- queue
- logs

Do not rebuild working pieces if they can be wrapped and reused.


STEP 4
ELEVENLABS VOICE GENERATION

Create production audio from finished scripts.

Output folder:

content_factory/audio/

Each script -> one voice file.

VOICE_MAP.json remains a core config file.


STEP 5
SUBTITLE GENERATION

Convert final script text into SRT subtitles.

Output folder:

content_factory/subs/

Initial version can be sentence-split timing.
Later version can use better word timing or alignment.


STEP 6
VIDEO RENDERING

Use ffmpeg through deterministic shell scripts.

Pipeline:

script
+ voice
+ subtitles
+ image
=
video

Output folder:

content_factory/videos/


STEP 7
CHARACTER IMAGE LIBRARY

Generate reusable image sets.

Female podcast host
30 images

Male podcast host
30 images

Psychology narrator
30 images

Total images:
90

Saved to:

images/female_host
images/male_host
images/psych_host

These reusable assets allow hundreds of videos without new image generation every run.


STEP 8
UPLOAD QUEUE

Before full auto-publish, create a queue-based upload layer.

Each finished video should generate upload tasks for:

- TikTok
- YouTube Shorts
- Instagram Reels

Queue entries must track:

- platform
- file path
- title
- caption
- hashtags
- status
- retry count
- last error


STEP 9
PLATFORM UPLOADERS

Future uploader modules may exist as scripts, APIs, or separate services.

Names may still include:

- tiktok_uploader
- youtube_shorts_uploader
- instagram_reels_uploader

But in the new architecture they are execution units behind n8n, not primary orchestrators.


STEP 10
REPORTING + FAILURE RECOVERY

Daily report should include:

- topics generated
- scripts generated
- audio files generated
- videos rendered
- uploads completed
- failures by stage
- queue backlog

A failed job should move to failed queue with reason, not disappear silently.


--------------------------------------------------
22. WHAT SHOULD BE REMOVED OR DOWNGRADED
--------------------------------------------------

The following ideas should be removed or downgraded:

1) OpenClaw as the main always-on production scheduler
2) Heavy reliance on agent autonomy for routine repeatable steps
3) Public internet exposure of local tools at the beginning
4) Scattered business logic across too many ad hoc shell commands
5) Full-scale 45 uploads/day before the pipeline is stable

These are not banned forever.
They are simply wrong as the starting architecture.

25. MASTER RULES FOR THIS PROJECT
--------------------------------------------------

1) Reuse what already works
2) Put repeatable logic into n8n
3) Keep MLX local and cheap
4) Use GPT / Codex only where quality or complexity justifies cost
5) Keep secrets out of prompts
6) Make every stage resumable
7) Prefer queues over fragile one-shot execution
8) Prefer stable output over clever architecture
9) Keep builder tools separate from runner tools
10) Speed matters, but recoverability matters more than elegance


--------------------------------------------------
26. FINAL STRATEGIC SUMMARY
--------------------------------------------------

This is not an OpenClaw-only content factory anymore.

This is a hybrid automation system.

OpenClaw builds and improves the system.
n8n runs the system.
MLX handles cheap local generation.
GPT / Codex handles higher-value thinking and coding.
